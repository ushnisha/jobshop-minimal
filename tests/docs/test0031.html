Test 0031:

  • This test has the two demands, both of them for the same part, PART001, and both due on 2018-01-14, and both for 270 units.
  • The demand for CUST001 has a priority of 1, while the demand for CUST002 has a priority of 2. That is, the demand for CUST001 has a higher priority and is planned first.
  • The RESOURCE_CONSTRAINED planning parameter is set to TRUE.

Variations from the Base Scenario:

  • Workcenter availability is a "hard constraint".
  • A ReleasedWorkOrder of 270 units for PART001-OP20 loads RES003 between 2018-01-06 and 2018-01-07. It is not assigned to any specific demandID.
  • Both demands are for the same part, PART001

The expected result is as follows:

  • Both demands are satisfied on time.
  • We can follow the constrained planning logic of by working through the steps of planning the demands, one at a time.
  • First we plan the SHIP for DMD001 (higher priority demand and is planned first). It needs to start on 2018-01-07 to complete on time.
  • PART001-OP20 for DMD001 is not planned afresh; instead the solver uses the ReleasedWorkOrder completing on 2018-01-07 for the same quantity as the demand (270). As a result, we don't need to plan PART001-OP10 for DMD001. Since the ReleasedWorkOrder for PART001-OP20 for DMD001 is on the exact same date and quantity as required for JIT planning the demand for DMD001 is met on time.
  • Next we plan the SHIP for DMD002. It needs to start on 2018-01-07 to complete just in time. As a result PART001-OP20 for DMD002 is planned to start on 2018-01-05 - note that this is a day earlier than PART001-OP20 for DMD001 - this is because RES003 is already reserved for PART001-OP20 for DMD001 on 2018-01-06 (the ReleasedWorkOrder for this Task). Consequently, PART001-OP10 for DMD002 is also pulled in earlier by one day and now starts on 2018-01-04.
  • DMD002 is delivered early since the solver tries to minimize inventory at every stage.
